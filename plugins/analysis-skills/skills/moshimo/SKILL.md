# moshimo

プロダクト/データの仮説を、思い込みや未検証の一般化ではなく **実測された結論** に落とすための調査規律。

## 聞き出し（本質：軸の定まった elicitation）

仮説構築の本質は **軸の定まった聞き出し**。データより先に、ブレない軸で問い、仮説を実測可能な形へ昇華させる。

**軸（全質問はこの観点のどれかに紐づく。紐づかない質問はしない）**：

1. 本質目標 — stated goal は proxy か／真の目標は何か
2. 出所/経緯 — どの観測・文脈から生まれたか（逸話か接地データか）
3. 解釈フォーク — 曖昧語を分岐に展開。フォークで測る対象/施策/指標が変わるなら確定させる
4. load-bearing 前提/リンク — 意思決定を実際に左右するのはどれか
5. 示唆/意思決定関連度 — その答えは次の行動を変えるか（変えないなら聞かない）
6. 測定定義/KPI — 「強い/効く/多い」を何の指標でどう測るか

**聞き出しの規律**：

- わからなければ仮定せず問う。ユーザの知識/熟達度に関係なく process を推し進める。
- load-bearing 軸から先に問う（答えやすい/安い質問に流れない）。
- 回答が軸を超える示唆を含んだら拾って仮説を昇華させる（新情報で問いを組み替える）。
- 軸に無い脱線には乗らない。軸を固定したまま核へ戻す。

**態度（内部制御。振る舞いで示し、posture 名や判定信号を出力に書かない）**：

- 不明 → 問う（上記軸で埋める）
- 目標非整合 / load-bearing リンク不在 → バッサリ切る（理由は最小限、深追いしない）
- 例外的に優れた仮説 → 称賛し、その考案の **必要条件を完全抽出**（漏れなく列挙）。判定は気分でなく軸の充足で
- 本質からの脱線 → 触れずに冷たく
- 出力は読み手優先で簡潔に。スキルの規約・軸・ルールを読み上げない（過程は振る舞いで示す）

## 開始時（常に）

依頼を 4 枠に分解して返す：

1. 仮説の妥当性（検証にとって妥当か）— **測れるかの前に、上記「聞き出し」の軸で仮説をクオリファイする**（下記 stop-and-check #0）
2. 既存データで測れるか
3. 測れるなら抽出して論じる
4. 測れないなら、必要な計装と工数（実装・保守）

主張が因果連鎖なら、各リンクを「妥当性 × データ有無」で **個別に** 評価する。

## 作業中の stop-and-check（言いかけたら止まる）

0. **データ/計装を取りに行きかけた、または成果物（ドラフト・複数案・結論・提案）を出しかけた** → 止まる。測る/作る前に上記「聞き出し」の軸で仮説を **クオリファイ**（本質目標との一貫性／出所／示唆／解釈フォーク）。そして **load-bearing なリンク（意思決定を実際に左右する）を特定** し、そこから測る。**安いリンク ≠ 重要なリンク**。一番安く測れるリンクに飛びつかない。**複数案を並べたくなったら背景が欠けているサイン** — 案を増やさず先に問う。

1. **「大きい/小さい/重要/効く」と言いかけた** → 絶対数・pp・月次件数・Fermi 上限に変換する。上限でも非本質的なら「その施策は追わない」判断まで出す。

2. **「たぶん/大半は/おそらく〜だろう」を数字なしで言いかけた** → 実測する。測れないなら断定せず **「不明＋測る方法」** を出す。（"assert" ではなく "measured or unknown"）

3. **あるレバーの効果を結論しかけた** → 相関と因果を分離する。交絡を名指しし、それを固定した **最小の統制比較** を設計する（例：プラットフォームを固定して速/遅を比較）。「証明」と言えるのは実験（AB）のみ。

4. **「データに無い/測れない/もう論点は残っていない」と言いかけた** → 在り処を疑う。client log / DB / 生アクセスログ / 外部基盤（メール・Braze 等）を triangulate し、結合キーを特定する。**split を信じる前に必ず陽性対照**（既知結果のコホートが ~100% 再現するか）で手法を検証してから本番の分離に進む。

5. **「後で/誰かが/将来やる」と注記に逃げかけた** → 今取れるかを確認する（proxy 指標・成熟済みの旧コホート・別データソース）。取れるなら **このターンで取る**。注記で他人任せにしない。

6. **「完了/やることは無い」と宣言しかけた** → 選択肢空間を列挙し、各項目を「実測 or 明示的に不明」に振り分ける。1 件の null 結果から「何も無い」を一般化しない。さらに、その結論が乗っている **load-bearing な前提を名指し** し、前提が未検証ならその結論は「不明」に格下げする。

7. **数値/外部主張を根拠に使いかけた** → 期待値（インパクト × 確度）で扱いを決める。データが在ること自体は主張の真偽を保証しない（欠測・偏り・粒度で「在るのに誤る」）。確度は入口でタグ付けし、高インパクト × 低確度は捨てず検証へ回す／低インパクトは未検証と明記して進む。

8. **ユーザが条件付けたスコープ（「〜において」等）から広げ/ずらしかけた** → 元のスコープ内を尽くしてから、拡張は明示的に提案する。無断で既探索の広域へ戻らない。

9. **施策/レバーを提案しかけた** → その自由度を先に確認する（既に議論され確定済みの制約ではないか）。確定済みなら提案せず、変更可能な余地に絞る。

10. **分散・残差・ギャップを単一要因に帰属しかけた** → 競合する分解要因を全て名指しし、手元データの粒度でそれらが識別可能かを確認する。識別不能なら帰属せず **「合成量の上界」** とだけ述べ、識別に必要なデータ粒度を明記する。帰属に乗せた下流の結論は帰属と一緒に格下げする。

## 終了時

- 全ての既存結論と整合を取り直す（矛盾・上書き済みの推奨を排除、サマリは実測に忠実な言い換えにする）。「不要」結論なら「次にやること」に作業メニューを残さない。
- 担当外（別チーム/別領域）はスコープを切って申し送りにし、深追いしない。

## 自己適用（再帰改善の手順）

- 是正・指摘を受けるたびに、同 dir の `LEDGER.md` へ 1 行追記し、**当時稼働していた版で catch できたか** を判定する。miss は最小のルール追記に落とし、skill-update で repo に反映する（規律追加 = minor bump）。
- **`LEDGER.md` はローカル限定**（事例に社外秘の数値/固有名を含むため、公開 repo へ同期しない。skill-update 側で exclude 済み）。repo に載せるのは抽象化されたルール本文のみ。
- スキル自身が生む失敗（例: 規約の読み上げ）も台帳の対象。中核原則は補助ファイルに埋めず本文へ。
- 有効性の現況: 制定後の是正 **n=1・miss 1 件**（07-08 #23: 残差の単一要因帰属 → v0.6 で#10 新設）。制定後の **catch は n=0** のままで、汎化は依然未証明。catch/miss が増えたらこの行を更新する。
