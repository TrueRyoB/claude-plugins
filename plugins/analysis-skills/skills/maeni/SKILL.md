# maeni（前に）

判断を実行の**前に**倒し、タスクを**前に**進める実行規律。タスクを done 定義・スコープ外・所有権つきの最小単位（単位カード）に分解し、実行中のその場判断を排除して並列に解く。

中心原理: 実行中の判断は「固定位置ゲート」と「想定外テーブル」だけに封じる。実行中に判断が必要になった時点でそれは段取りの失敗 — 停止・報告して段取りへ差し戻す。判断問題を機構問題に変換する。

## 1. 発注書（入力契約）

実行を開始する前に、以下 5 フィールドを確定させる。欠落があれば実行せず聞き出す。

```
- 本質目標: <1文。この run が何のためか>
- 土台: <前提事実のリスト。各行に [実測] or [未検証] タグ必須>
- 制約: <締切 / 触ってよい範囲の外周 / 予算>
- エスカレ先: <停止・報告の宛先と手段>
- 単位: <単位カードのリスト。無ければ段取りで作る>
```

入力ソースの優先順位:

1. **moshimo の出力**: 実測された結論 → 土台へ [実測] タグで写像。「次にやること」→ 単位候補。moshimo を通った前提だけが [実測] を名乗れる（interface の実体はこのタグ）
2. ユーザの直接発注
3. どちらも無ければ聞き出しで埋める

[未検証] 前提に乗る単位は停止条件を 1 段厳しくする（前提が崩れた兆候 = 即停止・報告）。

## 2. 段取り（実行前・一度だけ）

1. 発注を単位カードに分解する
2. **サイジング規則**: done が単一の検証コマンドで判定でき、所有 glob が他単位と非交差になるまで分解する。満たせなければ分解し直すか発注元へ差し戻す（実行で吸収しない）
3. 依存グラフを明記する。循環は設計エラー
4. 全単位の所有 glob の非交差を機械検査する。交差 → 単位統合 or 依存関係化
5. ディスパッチ計画: 並列起動できるのは「依存なし ∧ 所有非交差 ∧ リポジトリ状態が分離されている（別リポ / worktree）」のみ。分離できなければ直列化する

### 単位カード（中核 artifact）

```
### 単位 <ID>: <名前>
- done: <検証コマンド + 期待出力>（「〜できている」等の観測不能表現は禁止）
- スコープ外: <やらないことを最低 1 項目>
- 所有: <書き換えてよい glob。これ以外は読み取り専用>
- 依存: <先行単位 ID。無ければ ->
- 予算: <同一アプローチの試行上限。デフォルト 2>
- 想定外時: <デフォルト表 or 個別上書き>
```

## 3. 実行ループ（単位ごと・機械的）

この節と次節は弱い実行者がそのまま従う中核。解釈しない。実行する。

1. **ゲート A（単位開始の直前・必ず）**: done / スコープ外 / 所有を再読する。1 つでも解釈が二通りある → 開始しない。停止・報告する
2. 実行する。所有 glob 外は読み取り専用。変更したくなっても触らない
3. **ゲート B（完了宣言の直前・必ず）**:
   1. done の検証コマンドを実行し、出力を貼る。満たさないのに「実質完了」と言わない
   2. `git diff --stat` を所有 glob と突合する。glob 外の変更が 1 ファイルでもある → 停止・報告
   3. アンカー一問を 1 行出力する:「この結果は発注書の本質目標と整合しているか。Yes/No」。No → 停止・報告
4. **予算ゲート（自動）**: 同一アプローチの試行が予算を超えた → 即停止。「あと一回」は禁止
5. 報告して次の単位へ

停止・報告テンプレ（固定）:

```
単位<ID> 停止 / 事象: <1行> / 試行済み: <箇条書き> / 提案: <1つだけ>
```

## 4. 想定外テーブル（事前判断ルール）

実行中に想定外へ遭遇したら、この表を引く。表で決める。その場で考えない。

| 事象                                | 行動                                              |
| ----------------------------------- | ------------------------------------------------- |
| コマンド/テスト失敗                 | 同一アプローチ最大 2 回再試行。3 回目は停止・報告 |
| done の解釈が複数ありうる           | 停止・報告（推測で選ばない）                      |
| 所有 glob 外の変更が必要と判明      | 触らない。停止・報告（スコープ拡大は絶対禁止）    |
| 依存単位の成果物が期待と違う        | 自分で直さない。依存元 ID を名指しで停止・報告    |
| 改善アイデア/ついで修正を思いついた | 実行しない。「申し送り」に 1 行記録して続行       |
| 環境問題（ネットワーク/権限）       | 1 回再試行 → 停止・報告                           |
| **表にない事象**                    | **停止・報告（続行しない）** ← catch-all          |

禁止語彙（本スキル配下の全出力・全カード）:「適宜」「必要に応じて」「基本的に」「原則」。

## 5. 並行運用

- 共有ファイル（package.json / lockfile / schema 等）は所有者を 1 単位に固定する。他単位は変更要求を申し送りで出す
- **同一リポジトリの同一 worktree で複数単位を並列起動しない**（ゲート B の `git diff --stat` 突合に他単位の変更が混ざり誤停止する）。並列は別リポ / worktree 分離のみ。分離できなければ直列化する
- 機構はハーネスへ委譲する: 単位の追跡は TaskCreate、並列起動は Agent（1 メッセージで同時呼び出し）、schema 強制が要る箇所は Workflow、リポジトリ状態の分離が要る単位は EnterWorktree / isolation: worktree
- worktree 分離 run は末尾に必ず「統合単位」（merge + 全体 done 再検証）を 1 つ置く
- ワーカー起動: 同梱の `agents/maeni-worker.md` を `~/.claude/agents/` にコピーして使う（単位カード 1 枚 = ワーカー 1 起動）。ワーカーには単位カード・発注書の本質目標・想定外テーブルの 3 点だけを渡す

## 6. 終了時

1. 発注書との突合: 全単位の done 出力と本質目標を並べ、未達・glob 外変更・矛盾が無いか確認する
2. 申し送りを 1 か所に集約して報告する（ついで修正候補・glob 外の変更要求・改善アイデア）
3. RUN 台帳へ 1 行追記する（LEDGER.md セクション 2。実行中には記録しない）

## 7. 測定と現況

- **無介入完遂** = 発注確定後 run 終了までに、ユーザの**是正的**介入（方向修正・巻き戻し・書き直し指示）が 0 件
- 規律通りの停止・報告で得た回答は介入に**数えない**（数えるとエスカレしない方向へインセンティブが歪む）。頻度は RUN 台帳の「停止・報告回数」列で可視化し、トレードオフを観測可能にする
- 対照 run（bare = 本スキルなし）も同じ RUN 台帳に `条件` 列で記録し、KPI「無介入完遂率のハーネス有無比較」を成立させる
- 有効性の現況: 「ハーネスで弱いモデルの実行機能（目的維持・状況判断）を代替できる」は未証明命題。エビデンスは陽性対照（罠 2 種を規律通り処理）どまりで、実運用の catch/miss は **n=0**。RUN 台帳が埋まるまで底上げ効果を主張しない。埋まったらこの行を更新する

## 8. 自己適用（再帰改善の手順）

- 是正・指摘を受けるたびに、同 dir の `LEDGER.md`（セクション 1）へ 1 行追記し、**当時稼働していた版で catch できたか** を判定する。miss は最小のルール追記に落とし、skill-update で repo に反映する（規律追加 = minor bump）
- **`LEDGER.md` はローカル限定**（是正事例・RUN 記録に社外秘の数値/固有名を含み得るため、公開 repo へ同期しない。skill-update 側で exclude 済み）。repo に載せるのは抽象化されたルール本文のみ
- 中核原理（判断の前倒し・固定位置ゲート・想定外テーブル）は補助ファイルに埋めず本文へ
